## Введение

Документ описывает план тестирования веб‑приложения **Cinema Web App**. Цель тестирования – проверить соответствие реализованных функций требованиям, корректность работы фронтенда и бэкенда, выявить дефекты до релиза.

Основание: анализ исходного кода фронтенда (React: `src/App.js`, страницы `Home.js`, `MovieDetails.js`, утилиты `BuyTickets.js`, `Login.js`, `Register.js`) и бэкенда (Spring Boot: контроллеры `UserController`, `OrderController`; сервис `UserService`; репозитории `UserRepository`, `OrderRepository`).

Ключевой функционал в зоне тестирования: регистрация/логин пользователя, просмотр фильмов, заказ билетов, корректная обработка ошибок, уведомления.

<a name="items"/>

## Объект тестирования

**Фронтенд:** SPA на React с компонентами для просмотра фильмов, деталями фильма, формами логина/регистрации, системой уведомлений (`react-toastify`).
**Бэкенд:** Spring Boot REST API.

Основные REST‑эндпоинты:

* `/api/v1/register` — регистрация пользователя.
* `/api/v1/login` — вход пользователя.
* `/api/v1/order` — создание заказа, получение последнего заказа пользователя.

<a name="quality"/>

## Атрибуты качества

1. **Функциональность**

   * Полнота: регистрация, логин, создание заказа, получение последнего заказа.
   * Корректность: API возвращает ожидаемые коды и тела ответов; фронтенд отображает данные и ошибки.
2. **Удобство использования**

   * Интуитивный интерфейс: формы с валидацией, уведомления об ошибках через toast.
   * Основные операции (регистрация, логин, заказ билета) выполняются без лишних шагов.
3. **Надёжность**

   * Целостность данных при валидных и невалидных входных данных.
   * Сообщения об ошибках корректно отображаются пользователю.
4. **Производительность**

   * Время ответа API ≤ 2 сек для основных операций.
   * Асинхронные действия (например, обработка заказа) не блокируют UI.
5. **Доступность**

   * Стабильная работа фронтенда и API в течение 24/7 в тестовом окружении.

<a name="risk"/>

## Риски

* Ошибки синхронизации при параллельных заказах одного места.
* Некорректная обработка невалидных данных (например, отрицательные номера мест).
* Потеря уведомлений при сбое сети.
* Потенциальная несовместимость версий фронтенда и бэкенда.

<a name="features"/>

## Аспекты тестирования

### 1. Пользователи (`/api/v1/register`, `/api/v1/login`)

* Регистрация нового пользователя с валидными данными.
* Валидация: пустые поля, неверный email, слабый пароль.
* Логин с верными и неверными данными (401 Unauthorized).
* Сетевые ошибки и недоступность сервера → уведомление через toast.

### 2. Заказы (`/api/v1/order`)

* Создание заказа с корректными данными (userId, movieId, seat).
* Проверка seat ≥ 0 и корректного увеличения на фронте (`seat+1`).
* Ошибки: заказ для несуществующего фильма/пользователя → 400/404.
* Сетевые ошибки → уведомление через toast.
* Получение последнего заказа пользователя (`GET /api/v1/order/{userId}`).

### 3. Фронтенд (React)

* Корректный рендер страниц Home, MovieDetails.
* Формы регистрации и логина: валидация полей, блокировка кнопки при сабмите.
* Уведомления через `react-toastify` отображают успешные действия и ошибки.
* Заказ билета: отображение данных о фильме, номере места, времени и цене в toast.
* Работа навигации и компонентов NavBar/Footer.

<a name="nonfunctional"/>

## Нефункциональные требования

* UI корректно отображает ошибки сервера (без «сырых» сообщений).
* Время отклика списка фильмов ≤ 2 сек, заказ билета ≤ 1 сек.
* Доступность API ≥ 99% в тестовой среде.

<a name="approach"/>

## Подходы к тестированию

* **Ручное функциональное тестирование** через браузер.
* **Инструментальное тестирование API** (Postman/HTTPie/cURL).
* Для каждого сценария фиксируются: входные данные, шаги, ожидаемый и фактический результат, статус Pass/Fail.
* Негативные тесты: неверные данные, сетевые ошибки, пустые поля.

<a name="env"/>

## Окружение и данные

* Бэкенд: Java 17, Spring Boot; профиль dev.
* БД: MySQL локально, тестовые данные пользователей и фильмов.
* Фронтенд: Node.js LTS, сборка dev (`npm start`).
* Тестовые данные:

  * 20 пользователей, 50–100 фильмов, 30–50 заказов.

<a name="acceptance"/>

## Критерии приёмки

* Регистрация и логин пользователей работают корректно.
* Создание заказа проходит с верными данными; ошибки обрабатываются и отображаются пользователю.
* UI корректно обновляет состояние без перезагрузки страницы.
* Нет критических или блокирующих дефектов.

<a name="pass"/>

## Представление результатов

Результаты оформляются в документе **“Результаты тестирования”** (`Test_results.md`) с перечнем сценариев, шагов, ожидаемых/фактических результатов, логов и статусом.

<a name="conclusion"/>

## Выводы

План покрывает ключевые функции **Cinema Web App** и критические нефункциональные аспекты.
Прохождение тестов подтверждает корректность работы в рамках требований; обнаруженные дефекты фиксируются и повторно проверяются.
